# 阅读器放大与适配清晰度问题说明

## 背景

阅读器曾出现“质量模式下仍然不像原画质”的反馈。排查确认主阅读页已经取消 `DecodePixelWidth` 降采样，但视觉仍然偏软，说明问题不只在图片解码，而在显示链路。

需要区分三个概念：

- 原图解码：内存中拿到完整像素数据，不在加载阶段丢失细节。
- 原始显示：图像像素与屏幕物理像素 1:1 映射。
- 适配显示：为了让整页适应屏幕宽度或高度，对原图做缩小或放大显示。

“原图解码”不等于“屏幕上就是原画质”。只要显示阶段发生缩放，就会进入重采样。

## 现象

用户主要感知为：

- 阅读器适高/适宽后，线条、网点、文字边缘比原生阅读器软。
- 原图很大时，整页适配虽然能完整显示，但细节密度降低。
- Windows 高 DPI 缩放下，即使缩放条显示 100%，也可能不是物理像素 1:1。

## 根因

### 1. WPF 使用 DIP，而不是物理像素

WPF 的 `Width` / `Height` 单位是 DIP(device-independent pixel)，不是屏幕物理像素。

例如系统缩放为 150% 时：

- `Image.Width = 3000` 代表 3000 DIP。
- 实际屏幕像素约为 4500px。
- 如果原图是 3000px 宽，这会导致 WPF 把图像放大到 4500px 显示。

结果是：看似 100%，实际已经插值放大，必然不是真正原图像素。

修正方式：

```csharp
displayDipWidth = bitmap.PixelWidth / dpiScaleX;
displayDipHeight = bitmap.PixelHeight / dpiScaleY;
```

这样 `Zoom = 1` 才能接近屏幕物理像素 1:1。

### 2. 适高/适宽本质一定会缩放

适高/适宽的目标是“把整页塞进当前视口”。如果原图高于屏幕阅读区域，例如：

- 原图高度：3200px
- 阅读区域高度：1200px

那么适高必然做缩小采样。缩小后，多个原图像素会合并成一个屏幕像素，细线、网点、纹理和字边都会被重采样。

所以：

- `Original` 解决像素级查看。
- `Width` / `Height` 解决整页阅读便利。
- 两者不应该混为同一种质量标准。

### 3. WPF `BitmapScalingMode.HighQuality` 对漫画偏软

WPF 的 `HighQuality` 主要是通用高质量缩放，等价于 Fant。它适合普通照片，但漫画图像常见高频内容：

- 黑白线稿
- 网点
- 小字
- 高对比边缘

这些内容缩小时更需要边缘保持和适度锐化。单纯依赖 WPF 实时缩放会偏软。

### 4. 固定页槽会造成二次缩放

早期阅读器使用固定页槽，例如把页面先规范化到 `1600` 高，再由外层 `ScaleTransform` 做适配。这会让高分辨率图片先被压到统一虚拟尺寸，再按视口缩放。

该路径没有被删除，仍保留给性能模式使用。质量模式下会绕过固定页槽，改用原图像素作为布局基准。

## 当前实现策略

阅读器当前不是三条互斥显示路径，而是两个正交设置共同决定最终显示行为。

### 1. 解码质量：`ReaderQualityMode`

`ReaderQualityMode` 控制图片进入内存时的解码策略。

- `Quality`：当前页原图解码，不设置 `DecodePixelWidth`。
- `Performance`：按视口和缩放估算 `DecodePixelWidth`，降低内存与解码压力。

因此，“是否原图解码”只由 `ReaderQualityMode` 决定。

### 2. 布局方式：`FitMode`

`FitMode` 控制图片如何放进阅读器视口。

- `Width`：适宽。
- `Height`：适高。
- `Original`：缩放值设为 1。

因此，“是否适配视口”只由 `FitMode` 决定。

### Quality + Original

目标：真实 1:1 查看。

策略：

- 当前页原图解码。
- 不设置 `DecodePixelWidth`。
- 使用 `PixelWidth / DpiScaleX` 与 `PixelHeight / DpiScaleY` 计算 DIP 尺寸。
- `Zoom = 1`。
- 开启 `UseLayoutRounding` 与 `SnapsToDevicePixels`。

注意：这里的真实 1:1 指 `ReaderQualityMode.Quality + FitMode.Original`。如果切到 `Performance` 后再选 `Original`，仍会走性能模式的降采样和固定页槽路径，不是原图像素 1:1。

适用场景：

- 检查原图细节。
- 对比画质。
- 阅读局部小字或网点。

缺点：

- 大图无法整页显示，需要滚动。

### Quality + Width/Height

目标：整页阅读时尽量接近原生阅读器的清晰度。

策略：

- 先即时显示 WPF 高质量缩放结果，保证交互不断。
- 后台从原图生成当前视口尺寸的适配图。
- 对适配图做轻量 4 邻域 Laplacian 锐化，锐化强度按缩放比例动态调整。
- 生成完成后，如果当前页、当前模式、当前尺寸仍匹配，则替换显示。
- 适配图进入 LRU 缓存，缓存同时受数量与像素内存预算限制。
- 快速翻页、窗口尺寸变化、切换页面时取消过期任务。
- 当缩放比 `scale >= 0.98` 时，不启动后台适配图生成，直接使用 WPF `ScaleTransform` 实时渲染。
- 适配图生成会记录页码、缩放比例、结果内存、锐化强度、耗时和当前适配缓存占用。

该方案的核心思想是：不要每帧实时缩放原图，而是把“缩放结果”变成可复用位图。

优点：

- 翻页和调整时先有即时反馈。
- 后台完成后视觉更清晰。
- 同一页同一尺寸不重复计算。

缺点：

- 首次进入某个适配尺寸时会先软后清晰。
- CPU 仍参与重采样和锐化。
- 极大图在低性能设备上仍可能有后台计算压力。

当前锐化不是完整 3x3 卷积核，而是十字形 5 点 4 邻域 Laplacian：

```csharp
var edge = center * 4
    - original[i - 4 + c]
    - original[i + 4 + c]
    - original[up + x * 4 + c]
    - original[down + x * 4 + c];
```

它只采样左、右、上、下四个邻居，不采样四个角点。

### Performance + Width/Height/Original

目标：低内存、快翻页。

策略：

- 使用 `DecodePixelWidth` 受控降采样。
- 保留固定页槽和轻量显示路径。
- `NormalizePerformanceImageSizing` 使用 `FixedPageSlotHeight = 1600`。
- 适合极大图、低内存设备或快速翻页场景。

缺点：

- 牺牲部分细节。
- 不作为画质对比基准。
- 即使 `FitMode = Original`，也不代表原图像素 1:1。

## 为什么原生阅读器的适高可能更清晰

优秀原生阅读器一般不会只做普通实时缩放。常见策略包括：

- 使用 Lanczos / Mitchell / HighQualityCubic 等更适合缩小的采样算法。
- 缩小后做轻量 unsharp mask。
- 为当前视口生成缓存位图。
- 使用 GPU 或专门图像库进行快速重采样。
- 对漫画线稿和网点做专门参数调校。

因此，适高本身不是原画质，但可以通过更好的采样和锐化，让整页阅读观感更接近原图。

## 已知边界

当前方案仍有边界：

- 锐化强度已按缩放比例动态调整，但尚未暴露为用户配置。
- 后台适配图使用 CPU 生成，尚未引入 SkiaSharp / WIC COM / GPU 管线。
- WPF `Image` 控件仍负责最终显示，显示层仍受 WPF 渲染体系约束。
- 适高/适宽是缩小结果，不可能保留所有原图像素细节。
- 放大超过 100% 时不会产生真实新细节，只会插值放大。
- `Performance + Original` 不是真正 1:1，只是性能模式下缩放值为 1。

## 后续优化方向

### 方向 1：继续优化当前 WPF 管线

可做：

- 对适配图缓存继续细化内存预算，例如按设备内存等级调整阈值。
- 继续调校缩放比例到锐化强度的映射。
- 区分彩色页、黑白页、网点页使用不同参数。
- 基于后台生成耗时日志建立真实性能数据。

优点：

- 改动小。
- 不引入新依赖。
- 发布包稳定。

缺点：

- WPF 图像显示天花板仍存在。

### 方向 2：引入 SkiaSharp 图像管线

可做：

- 使用 SkiaSharp 解码/重采样/绘制。
- 使用更成熟的采样器和滤镜。
- 后续可尝试 `SKGLElement` 做 GPU 加速。

优点：

- 图像处理能力更强。
- 更接近专门阅读器的渲染方式。
- 后续扩展锐化、色彩、滤镜更自然。

缺点：

- 增加 native 依赖。
- 发布包变大。
- Release 打包链路必须重新验证。

### 方向 3：直接使用 WIC/Direct2D

可做：

- 调用 WIC `HighQualityCubic`。
- 使用 Direct2D 渲染目标。

优点：

- Windows 原生能力强。
- 适合 Windows-only 高质量管线。

缺点：

- COM / interop 复杂度高。
- 维护成本明显上升。
- 与未来跨端壳不天然兼容。

## 当前推荐

短期继续使用当前 WPF 后台适配图缓存方案，通过现有性能日志观察真实大图场景下的耗时与内存。

如果用户仍明确感知不如原生阅读器，再引入 SkiaSharp 作为实验渲染后端，而不是继续在 WPF `Image + ScaleTransform` 上堆参数。

判断标准：

- 如果问题集中在适配缩小时偏软，优先优化适配图采样和锐化。
- 如果问题集中在大图切换卡顿，优先优化缓存、任务取消和预生成。
- 如果问题集中在缩放拖动实时性，考虑 SkiaSharp / GPU。
- 如果问题集中在 100% 不够清晰，优先检查 DPI 映射与像素对齐。
