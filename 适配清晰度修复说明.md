# 适配清晰度修复说明

## 问题

Quality 模式 + 适高/适宽，当漫画图片高度与屏幕物理像素高度接近时，用户感知到明显的画质损失，与预期的"近 1:1 几乎无损"不符。

## 根因分析

### 为什么 scale 不是 1.0

图片高度恰好等于 2K 屏物理高度（1440px）时，理论 scale 应为 1.0。但实际计算：

```
available = 视口高度 - ScrollViewer Padding(12) - Image Margin(8)
source    = PixelHeight / dpiScaleY = 1440 / 1.5 = 960 DIP

窗口模式（有标题栏）：视口约 920 DIP
available ≈ 920 - 20 = 900 DIP
scale = 900 / 960 ≈ 0.94
```

窗口 chrome 和内边距共消耗约 60px DIP，使 scale 从理论的 1.0 跌至 0.93–0.95。

### 原 bypass 阈值不足

原阈值为 `scale >= 0.98`：

- 0.94（窗口模式近 1:1）→ 进入质量适配流程
- CPU TransformedBitmap 重采样（1440px → ~1350px）
- 叠加固定 0.08 锐化 → 人工痕迹可见

## 修复内容

### 1. bypass 阈值：0.98 → 0.93

```csharp
// ReaderWindow.xaml.cs - TryApplyQualityFit
if (scale >= 0.93)   // 原来是 0.98
{
    CancelQualityFitRequest();
    ZoomSlider.Value = scale;
    return true;      // 直接用 WPF ScaleTransform，不生成适配位图
}
```

覆盖范围：

| 场景 | scale 约值 | 修复前 | 修复后 |
|---|---|---|---|
| 全屏，图片 ≈ 屏高 | 0.97–0.99 | 接近 bypass | bypass ✓ |
| 窗口模式，图片 ≈ 屏高 | 0.93–0.95 | 触发质量适配 | bypass ✓ |
| 图片明显大于视口 | < 0.93 | 触发质量适配 | 触发质量适配 ✓ |

scale 在 0.93–0.98 区间内，WPF 的 HighQuality（Fant）ScaleTransform 做 5–7% 的微缩，结果与原图视觉上几乎无差异，不需要额外处理。

### 2. 近 1:1 档锐化量：0.08 → 0.04

```csharp
// ReaderWindow.xaml.cs - GetSharpenAmount
private static double GetSharpenAmount(double scale)
{
    if (scale >= 0.85) return 0.04;  // 原来是 0.08
    if (scale >= 0.65) return 0.14;
    if (scale >= 0.45) return 0.18;
    return 0.22;
}
```

对于仍落入质量适配流程（scale 0.85–0.92）的场景，原 0.08 锐化量对漫画线稿会产生可见的人工锐边。0.04 在该缩放范围内仍有边缘补偿效果，但不会过冲。

完整锐化曲线：

| scale 范围 | 缩放幅度 | 锐化量 | 说明 |
|---|---|---|---|
| ≥ 0.93 | < 7% | — | bypass，不生成适配位图 |
| 0.85–0.92 | 8–15% | 0.04 | 极轻补偿 |
| 0.65–0.84 | 16–35% | 0.14 | 中度锐化 |
| 0.45–0.64 | 36–55% | 0.18 | 标准锐化 |
| < 0.45 | > 55% | 0.22 | 重度锐化 |

## 未改动部分

- 质量适配位图的生成逻辑、缓存结构、内存预算不变
- TransformedBitmap + 4 邻域 Laplacian 的核心流程不变
- Performance 模式逻辑不变
